# Specification Writer

This skill guides the creation of professional technical specifications and architecture documents.

## When to Use

Use this skill when you need to:
- Write technical architecture documents
- Define API specifications and interfaces
- Create system design proposals
- Document database schemas and data models
- Produce implementation guides for development teams
- Communicate technical decisions to stakeholders

## Document Types

### 1. Architecture Specification
Documents system structure, component relationships, and integration points.

**Structure:**
- Executive Summary
- System Overview
- Component Architecture
- Data Flow Diagrams
- Interface Definitions
- Deployment Architecture
- Security Considerations
- Performance Requirements

### 2. API Specification
Documents RESTful or GraphQL APIs with endpoints, request/response schemas.

**Structure:**
- API Overview
- Authentication & Authorization
- Base URL & Versioning
- Endpoint Reference
- Request/Response Schemas
- Error Handling
- Rate Limiting
- Code Examples

### 3. Technical Design Document (TDD)
Detailed design for a specific feature or component.

**Structure:**
- Problem Statement
- Goals & Non-Goals
- Proposed Solution
- Alternative Solutions Considered
- Data Model
- API Changes
- UI/UX Changes
- Testing Strategy
- Rollout Plan

### 4. Database Schema Documentation
Documents database structure, relationships, and constraints.

**Structure:**
- Schema Overview
- Entity Relationship Diagram
- Table Definitions
- Index Strategy
- Migration Strategy
- Backup & Recovery

## Writing Guidelines

### Clarity First
- Use clear, concise language
- Avoid ambiguous terms
- Define acronyms on first use
- Include concrete examples

### Structure Matters
- Use consistent headings
- Include table of contents for longer docs
- Break complex sections into subsections
- Use bullet points for lists

### Visual Aids
- Include diagrams for architecture (Mermaid preferred)
- Use tables for structured data
- Add flowcharts for processes
- Include code examples where relevant

### Version Control
- Include version number
- Track changes in changelog
- Date all versions
- Identify authors/reviewers

## Best Practices

1. **Audience Awareness**: Write for the intended audience (developers, stakeholders, ops)
2. **Completeness**: Cover what, why, and how
3. **Accuracy**: Verify all technical details
4. **Maintainability**: Structure for easy updates
5. **Review Process**: Include review checklist

## Review Checklist

- [ ] Technical accuracy verified
- [ ] All diagrams are clear and correct
- [ ] Code examples tested
- [ ] Links and references valid
- [ ] Consistent terminology throughout
- [ ] Approved by relevant stakeholders
